Opal Administrator Guide

Deployment, operations, and user management reference

Overview

Opal (Operational Priority and At-Risk Likelihood) is an internal web application for tracking HPE Networking customer escalation risk. Account managers and sales engineers submit weekly status via Microsoft Forms; a CSV export is ingested into Opal to populate the dashboard.

Key capabilities:

  • Heat-coded customer dashboard — Critical, Hot, Concerned, Stable
  • Per-customer detail and edit pages
  • Executive overview with regional breakdown
  • Stale record detection (top 20 longest without an update)
  • Secure login with role-based access (admin / user)
  • Week-over-week heat trend tracking with dashboard indicators
  • Email alerts when a new Critical customer is ingested
  • Full audit trail of every database change
  • Automated backups twice daily (6 AM and 6 PM)

The application runs as a single Docker container with a SQLite database persisted to the host filesystem.

Installation

1. Clone the repository

git clone https://github.com/xod442/opal.git
cd opal

2. Create the data directory

Opal stores the database and backups in ./data/ on the host. Docker Compose creates this automatically on first run, but you can also create it manually:

mkdir -p data/backups

3. (Optional) Set a production secret key

The default SECRET_KEY is fine for internal use but should be changed for any internet-facing deployment. Edit docker-compose.yaml and add the environment variable: 

environment:
  DB_PATH: /data/opal.db
  BACKUP_DIR: /data/backups
  SECRET_KEY: your-random-secret-here

4. Build and start

docker compose up -d --build

The first build downloads the Python base image and installs dependencies — this takes 1–2 minutes. Subsequent starts use the cached image and are nearly instant.

5. Verify it is running

docker compose ps

You should see the opal-opal-1 container with status Up. Open http://localhost:9090 in a browser — you will be redirected to the login page.

Resetting to factory defaults

To wipe all data and start completely fresh — for example, when redeploying to a new host or clearing a test environment:

Destructive. This permanently deletes the database, all customer records, users (including the admin account), audit logs, backups, and email settings. There is no undo. 
docker compose down
rm -rf data/
docker compose up -d

After the container restarts, Opal recreates the database with a fresh default admin / admin account and empty tables. Follow the First Login steps to set a new password.

Tip: If you only want to reset the database but keep your backups, move data/backups/ somewhere safe before deleting data/, then restore after the container comes back up.

Starting & Stopping

Start the application

docker compose up -d

Stop the application

docker compose down

The database is in ./data/opal.db on the host and is unaffected by stopping the container.

View live logs

docker compose logs -f

Rebuild after a code update

If you pull a new version from GitHub or edit any source files:

docker compose down
docker compose build --no-cache
docker compose up -d

Restart the container

docker compose restart
Auto-restart: The container is configured with restart: unless-stopped, so it automatically comes back up after a reboot or Docker restart.

Ingesting CSV Data

Customer data originates from a Microsoft Forms survey. Each week, the form owner exports the results to a CSV and ingests it into Opal. There are two methods:

Method 1 — Admin upload (recommended)

  1. Log in as an admin and click Admin in the dashboard header.
  2. Under Upload CSV, click Choose File and select the export file.
  3. Click Upload & ingest.
  4. A confirmation banner shows how many records were inserted, how many were duplicates, and how many Mist rows were skipped.

Method 2 — Command line (docker compose)

Place the CSV file in the ./csv/ directory on the host, then run:

mkdir -p csv
cp ~/Downloads/engagement.csv csv/
docker compose run --rm ingest

Ingest rules

RuleBehaviour
Deduplication Each row is keyed on the Microsoft Forms ID column. Re-ingesting the same CSV is safe — existing rows are silently ignored.
Mist filter Rows where the Current deployed Architecture field starts with "Mist" are skipped. These are a different product line and are not tracked in Opal.
Encoding Microsoft Forms exports with a UTF-8 BOM (utf-8-sig). Opal handles this automatically.

Backup & Restore

Automatic backups

Opal automatically backs up the database twice daily — at 6:00 AM and 6:00 PM server time. Backups are stored in ./data/backups/ as timestamped .db files (e.g., opal_backup_20260601_060000.db). The 20 most recent backups are kept; older ones are deleted automatically.

Manual backup

  1. Log in as an admin and click Admin.
  2. Under Backup Database, click Run backup now.
  3. The new backup appears immediately in the Restore from Backup table.

Restoring from a backup

Safety net: Before restoring, Opal automatically creates a backup of the current database. You can always undo a restore by restoring the backup that was created immediately before it.
  1. Log in as an admin and click Admin.
  2. In the Restore from Backup table, find the desired backup and click Restore.
  3. Confirm the dialog. The page reloads with a confirmation message.

Secondary backup location

You can configure a second destination for every backup — useful for copying to a mounted NAS, external drive, or any path accessible inside the container.

  1. Log in as an admin and click Admin.
  2. Under Backup Database → Secondary Backup Location, enter an absolute path accessible inside the container (e.g. /mnt/nas/opal-backups).
  3. Click Save.

Every subsequent backup — both scheduled and manual — will be copied to the secondary location in addition to the primary ./data/backups/ folder. If the secondary path is unreachable, the primary backup still completes normally. Clear the field and save to disable the secondary copy.

The path must be reachable inside the container. Mount the target volume in docker-compose.yaml under the volumes: key before using it here.

Backing up manually from the host

The database is a plain SQLite file. You can copy it directly:

cp data/opal.db data/opal_manual_backup_$(date +%Y%m%d).db

Exporting Data

The entire customer database can be exported to CSV at any time. The export includes all fields: customer name, heat level, at-risk status, risk reasons, architecture, near-term goals, BU contact, notes, and last modified date.

  1. Log in as an admin and click Admin.
  2. Under Export to CSV, click Download CSV.
  3. The file downloads as opal_export_YYYYMMDD_HHMMSS.csv.

Delete Database

The Delete Database function is available at the bottom of the Admin page. It permanently wipes all customer records, users, audit logs, heat history, and email settings, then immediately recreates a fresh empty database with the default admin account.

Irreversible. All data is permanently deleted. Run a manual backup first if you need to preserve anything.
  1. Log in as an admin and click Admin.
  2. Scroll to Delete Database at the bottom of the page.
  3. Type DELETE (all caps) in the confirmation field.
  4. Click Delete database.
  5. You are redirected to the login page. Log in with admin / admin and set a new password.

What happens after deletion

Opal immediately recreates the database with all tables and a fresh default admin account. You do not need to restart the container. The action is recorded in the new audit log before the old database is removed.

Note: Your session cookie becomes invalid after deletion because it references the old database. Opal redirects you to the login page automatically — log in with admin / admin and you will be prompted to set a new password.

Self-Registration

Users can create their own accounts without admin involvement. A Create an account link appears below the sign-in form on the login page.

How it works

  1. From the login page, click Create an account.
  2. Enter a unique username, an @hpe.com email address, and a password (minimum 8 characters).
  3. Click Create account.
  4. On success, you are redirected to the login page with a confirmation message and can sign in immediately.

Account level

All self-registered accounts are created at user level. They can view the dashboard and edit customer records, but cannot access the Admin page. An admin can promote the account to admin role via Admin → User Management if needed.

HPE email required. Only @hpe.com addresses are accepted. The domain is validated in the browser and again on the server — non-HPE addresses are rejected.
Self-registration is logged in the audit trail as a create_user action with the actor shown as self-register.

Managing Users

Only admin-role users can access the Admin page and manage accounts.

Creating a new user

  1. Go to Admin → User Management.
  2. Fill in the username, optional email, a temporary password, and role (user or admin).
  3. Click Create user.
  4. The new user will be required to change their password on first login.

Disabling / enabling a user

Click Disable next to the user in the User Management table. Disabled users cannot log in. Click Enable to restore access. You cannot disable your own account.

Resetting a password

Click Reset pw next to the user. Opal generates a random temporary password and displays it in the confirmation banner at the top of the Admin page. Give that password to the user — they will be required to set a new password on their next login.

Tip: There is no email-based password reset. You must relay the temporary password to the user out-of-band (Slack, Teams, etc.).

Password Policy

  • Minimum 8 characters.
  • Passwords are hashed with bcrypt — they are never stored in plain text.
  • Sessions are signed with HMAC and expire after 8 hours of inactivity.
  • Any account created by an admin, or any account that has had its password reset, must change the password on next login before any other action is allowed.
  • Users can change their own password at any time via the Pwd link in the dashboard header.

Detail & Edit

Detail page

Click Details → on any row to open the full customer record. All fields are shown read-only, including the Additional Notes section highlighted in blue.

Edit page

Click Edit on the dashboard row, or the Edit button on the detail page. The edit form is divided into two sections:

  • Read-only fields — Account Manager, Sales Engineer, Location, Submitted by, Submission date, and Customer Name. These come from the original form submission and cannot be changed.
  • Editable fields — Customer Temperature, Actively at risk, Risk reasons, Architecture, Near term goals, BU contact, Ask from business unit, Background, and Additional Notes.

Click Save changes to commit and return to the dashboard. Click Support to open the Support page for this customer. Every save is recorded in the Audit Log.

Support Page

The Support page is accessed via the Support button on the Edit page. It provides a focused view for tracking active engagement with a customer — separate from the core heat and risk fields so that support teams can update their fields without touching the original survey data.

Available at Trends in the dashboard header. Shows week-over-week heat movement for every customer — who is getting worse, who is improving, and who is new.

Trend table

The table lists every customer with their previous heat level, current heat level, when the change was recorded, and their AM and SE. Click the filter tabs to narrow to a specific direction. Click Details → to go directly to the customer's detail page.

Dashboard trend indicators

The main dashboard Heat column also shows inline trend indicators:

How snapshots are recorded

  • CSV ingest — a snapshot is recorded for each newly inserted customer.
  • Edit save — a snapshot is recorded whenever the heat level is changed on the Edit page.
  • Deduplication — if the heat level has not changed since the last snapshot, no new entry is written. Re-ingesting the same CSV does not create duplicate history.
Note: Trends are based on the two most recent heat snapshots per customer. A customer needs at least two snapshots (i.e., their heat must have changed at least once) to show a directional trend. Brand-new customers show as New until their heat changes.

Stale Records

Available at Stale Records in the dashboard header. Shows the 20 customers that have gone the longest without any update. Age is calculated from the last_modified field, falling back to the original submission date for records that have never been edited.

Rows are colour-coded: records untouched for 30+ days are highlighted amber; 60+ days are highlighted red. Use this page during weekly reviews to prompt the team to refresh stale entries.

Troubleshooting

Container won't start

docker compose logs

Look for Address already in use. If port 9090 is taken, change the left-hand port in docker-compose.yaml:

ports:
  - "9191:8000"   # change 9090 to any free port

Forgot the admin password

If you are locked out, run the following command from the host. It resets the admin password to admin and forces a password change on next login.

Important: Always run this command inside the container using docker exec. Running it with the host Python will fail silently due to a bcrypt version mismatch — the hash will be written but the password will not work. 
docker exec opal_opal_1 python3 -c "
from passlib.context import CryptContext
import sqlite3
pwd_ctx = CryptContext(schemes=['bcrypt'], deprecated='auto')
conn = sqlite3.connect('/data/opal.db')
conn.execute(\"UPDATE users SET password_hash=?, must_change_password=1 WHERE username='admin'\",
             (pwd_ctx.hash('admin'),))
conn.commit()
conn.close()
print('Done')
"

Log in with admin / admin — you will be prompted to set a new password immediately.

CSV upload produces 0 records inserted

  • All rows in the file may already be in the database (duplicate IDs). This is expected when re-ingesting the same export.
  • All rows may have been filtered as Mist rows. Check the Current deployed Architecture column in the CSV.
  • The CSV may be missing the ID column, causing all rows to be treated as ID 0 and colliding on insert.

Database was accidentally deleted

Act quickly. If the container is still running, the in-memory scheduler may write a new empty database. Stop the container first. 
docker compose down
cp data/backups/opal_backup_<latest>.db data/opal.db
docker compose up -d

Backups are not being created

The scheduler runs inside the container. Verify the container has been up continuously through 6 AM or 6 PM. If the container was restarted just before the scheduled time, it may have missed the window. Use Admin → Run backup now to create a manual backup at any time.

Test email fails

The error message from the SMTP server is shown in the admin banner. Common causes:

  • Authentication failed — double-check username and password. For Office 365, use your full email address as the username.
  • Connection timed out — the SMTP host or port is wrong, or a firewall is blocking outbound SMTP. Try port 465 if 587 is blocked.
  • Settings incomplete — host, username, password, and recipient are all required. Save settings before clicking Send test email.

All test email attempts and results are recorded in the Audit Log.

Session keeps expiring

Sessions are valid for 8 hours. If you are being logged out sooner, check that the container has not been restarted — a restart with a different SECRET_KEY invalidates all existing sessions. Set a fixed SECRET_KEY in docker-compose.yaml to prevent this.
